## Cross-cutting themes

1. **Calibrated prior `p_occ ≈ 0.933 ≠ 0.5`.** The split/unified substrates deliberately seed occupancy at `Beta(C·α₀, α₀)` (prior `p_occ = C/(C+1) ≈ 0.933` for NYU13 C=14), not `Beta(1,1)`. Thresholds, equivalence claims and visualisation gates that assume the old 0.5 prior misfire on prior-only voxels — findings **#1, #2, #16, #17, #32**.
2. **Missing lower clamp on `(C−K)·α₀` / prior subtraction.** Several consensus-merge and projection sites subtract the prior with no `max(0,·)`, admitting negative Dirichlet/Beta α → `p_occ` outside [0,1] → garbage entropy/EIG — findings **#4, #10, #23, #26**.
3. **Untrusted-wire hardening.** v4 deserialize trusts `num_classes`/`alpha_0` and per-voxel counts straight off the wire — findings **#3, #8, #9, #38**.
4. **Differential-entropy / Hutter numerical family** — findings **#5, #11, #12, #37, #40**.
5. **The new v4 RPC path (uncommitted at review time) has zero test coverage** and several query-math rough edges — findings **#16–#20, #30–#34, #46**.

### #1 — [`src/scovox_core/src/semdir_map.cpp:267`](src/scovox_core/src/semdir_map.cpp#L267)  ·  🔴 HIGH · CONFIRMED · numerical

**SemDir prior p_occ is ~0.93 (not 0.5), so the carve wall-guard mis-fires against the production default carve_skip_occ_threshold=0.7, treating prior-only voxels as walls and truncating carve rays.**

*Failure scenario.* defaultSemDirVoxel(C=14, alpha_0=0.01) gives alpha_free=0.01, alpha_other=12*0.01=0.12, cnt=[0.01,0.01]; p_occ = s_occ/s_total = 0.14/0.15 = 0.933. scovox_node.cpp:135 plumbs P.carve_skip_occ_threshold (default 0.7 from line 243) into SemDirMap::Params. Any already-allocated voxel still at (or near) the prior — e.g. a voxel allocated by applyHitUpdate with quality=0 (grazing/max-range ray: w_occ_share=0, class_share=0, nothing added, p_occ stays 0.933), or allocated then never carved — reads p_occ()=0.933 > 0.7 in applyCarveUpdate, returns false, and the carve loop in carveRay/the fused walker stops emitting a_free for the entire remainder of that ray. SemBeta is immune (its prior p_occ=0.5); only the SemDir substrate has this asymmetry. Result: spurious wall-blocking and under-carved free space along rays passing through prior-state voxels.

*Suggested fix.* Either gate the wall check so it only triggers once a voxel has accumulated real occupied evidence (e.g. require s_occ - other_prior_ > epsilon), or compute p_occ for the wall guard excluding the OTHER prior mass, or document/raise the SemDir carve_skip default and reject migrated 0.7 thresholds. Do not let prior-only voxels read as walls.

### #2 — [`src/scovox_core/include/scovox/scovox_map_split.hpp:165`](src/scovox_core/include/scovox/scovox_map_split.hpp#L165)  ·  🔴 HIGH · CONFIRMED · correctness

**Fused walker skips the semantic/occupancy hit update at the endpoint voxel whenever proj is ~0 (notably when the endpoint lands exactly on a voxel centre), while the non-fused path always applies it.**

*Failure scenario.* integrateHit(origin,endpoint,...) with endpoint exactly at a voxel centre (e.g. an axis-aligned synthetic point like (0.075,0.025,0.025) at res 0.05, or any glancing ray where (vc-origin)·(endpoint-vc)~0). In visit_one, c==k_hit gives v_point_voxel=endpoint-vc=0 so proj=0, the `if (std::fabs(proj) < 1e-12f) return;` fires BEFORE the `if (c == k_hit) semHit(...)` branch, so the surface voxel receives NO Stream-A occupancy bump and NO Dir class commit. The non-fused integrateHit (and SemDirMap::integrateHit) call applyHitUpdate(k_hit) unconditionally, so fused vs split diverge and the hit surface voxel is left at prior (p_occ unchanged, no semantics) for that ray.

*Suggested fix.* Move the k_hit semHit dispatch above the proj==0 early-return (handle the hit voxel before computing sign/sdf), or special-case dist<eps for k_hit to still run semHit. The TSDF band update can keep skipping on proj~0, but semHit must run for k_hit regardless.

### #3 — [`src/scovox_core/include/scovox/binary_serializer_v4.hpp:180`](src/scovox_core/include/scovox/binary_serializer_v4.hpp#L180)  ·  🔴 HIGH · CONFIRMED · api-misuse

**deserialize() reads num_classes and alpha_0 from untrusted wire input but never validates them (no finite/positive/>=K_TOP floor), and the downstream merge reconstructs the Dirichlet/Beta prior directly from these fields, so a corrupt header silently corrupts every merged voxel.**

*Failure scenario.* A v4 frame with num_classes=0 (or alpha_0=NaN / negative) passes deserialize() unchecked. In consensus_merge_v4.hpp mergeBetaV4 computes occ_prior = num_classes*alpha_0 = 0, so a_occ = a.a_occ + b.a_occ - 0 double-counts the prior; mergeDirV4 computes other_prior = (num_classes - K_TOP)*alpha_0 = (0-2)*alpha_0 < 0, inflating f.other. Two re-sent snapshots from the same corrupt sender agree with each other so mergeFramesV4's a.num_classes!=b.num_classes equality check (line 119) does NOT catch it, and NaN/negative mass propagates into the live map.

*Suggested fix.* In deserialize() (v3 and v4) after reading the header, validate: num_classes >= K_TOP, std::isfinite(alpha_0) && alpha_0 > 0; throw std::runtime_error on violation, same as bad MAGIC/VERSION.

### #4 — [`src/scovox_core/include/scovox/consensus_merge_v4.hpp:91`](src/scovox_core/include/scovox/consensus_merge_v4.hpp#L91)  ·  🔴 HIGH · CONFIRMED · correctness

**mergeDir/mergeSemDir/mergeSemBeta are not fold-order independent: top-K eviction plus a non-stable tie sort make the fused slots depend on source iteration order, contradicting the documented order-independence and producing a nondeterministic dominantClass.**

*Failure scenario.* K_TOP=2. Source a tracks classes {1:cnt5.0, 2:cnt5.0}, source b tracks {3:cnt5.0, 4:cnt5.0} (all four distinct, evidence tied at the K_TOP boundary). mergeDir(a,b) keeps {1,2} and dumps 3,4's evidence to OTHER; mergeDir(b,a) keeps {3,4} and dumps 1,2. So mergeDir(a,b) != mergeDir(b,a). refoldCellDir/refoldCellBeta (dscovox_node.cpp ~1194) folds sources by iterating sources_, a std::unordered_map whose iteration order is unspecified and changes on rehash; the seed-copy (first non-prior source copied verbatim, rest merged) makes the result depend on which source is visited first. A non-tied but truncation-crossing case is equally affected: class 9 with cnt4 in source u1 and cnt4 in source u2 (combined 8, would beat a third class at 7) sometimes wins a slot and sometimes is irretrievably merged into OTHER depending on fold order, because once evidence is folded into OTHER it can never climb back into a slot. Net effect: the fused semantics grid, dominantClass(), and the mesh labels can flip between runs/rehashes for the same input set. Mass is conserved but the per-class breakdown and argmax are not deterministic.

*Suggested fix.* Make the tie-break and slot-selection deterministic and order-invariant: sort with a stable secondary key (e.g. by (cnt desc, cls asc)) AND accept that single-pass union truncation cannot recover a class once it lands in OTHER. For true order independence the consensus must accumulate per-class evidence across ALL sources before truncating (e.g. refold by summing every source's slot+OTHER mass into a class->evidence dict, then truncate once), rather than pairwise merge-then-truncate. At minimum, document that fold order is NOT independent and key fold order on a sorted source id, not unordered_map iteration order.

### #5 — [`src/scovox_core/include/scovox/uncertainty.hpp:70`](src/scovox_core/include/scovox/uncertainty.hpp#L70)  ·  🔴 HIGH · CONFIRMED · numerical

**hutterEscapeMass divides by log(ratio) which approaches 0 as ratio approaches 1 from above, so effectiveResidual explodes to thousands when total semantic evidence N is small.**

*Failure scenario.* Voxel with one tracked class of small count and no residual, e.g. sem_cnt={0.0001,0}, a_unk=0 (or a_unk=0.0001 with all slots empty): estimateDistinctClasses=1, N approx 0.0001, ratio=(N+1)/1 approx 1.0001>1 so the `ratio<=1` guard is NOT taken, log(1.0001) approx 1e-4, and hutterEscapeMass returns 1/(2e-4) approx 5000. effectiveResidual then returns max(a_unk, ~5000)=~5000 — an escape mass far larger than the total number of observations, which is physically impossible (escape mass is bounded by N).

*Suggested fix.* Clamp the result: return std::min(hutter, N) (escape mass cannot exceed total observations), or guard ratio with a larger epsilon (e.g. if (ratio < 1.f + 1e-3f) return m;), or compute via std::log1p of the small quantity and cap the output.

### #6 — [`src/scovox_mapping/src/scovox_node.cpp:1024`](src/scovox_mapping/src/scovox_node.cpp#L1024)  ·  🔴 HIGH · CONFIRMED · memory-safety

**loadTopkForFrame computes the buffer size from header fields read without checking the reads succeeded; a truncated/empty .topk file yields a garbage size and crashes the node on resize().**

*Failure scenario.* With topk_probs_dir_ set, a frame whose 000000.topk file exists but is empty or shorter than the 5-byte header: f.read into uninitialized uint16_t H,W / uint8_t C all fail, leaving garbage. total = (size_t)H*W*C becomes enormous (e.g. ~1e15), topk_cache_probs_.resize(total) throws std::bad_alloc, which is uncaught in the integration callback → std::terminate kills the mapper. ### #7 — [`src/scovox_core/include/scovox/scovox_map_split.hpp:170`](src/scovox_core/include/scovox/scovox_map_split.hpp#L170)  ·  🟠 MEDIUM · CONFIRMED · correctness

**With space_carving=true the fused walker's `sdf <= trunc + h` pre-gate drops front-of-surface free voxels that both TsdfMap::integrateRay (non-fused) and the legacy scovoxmap walker integrate as the +trunc clamp, so the two TSDF paths produce different grids.**

*Failure scenario.* Set tsdf_space_carving=true (launch param, supported). Non-fused integrateHitSplit calls tsdf_.integrateRay which walks [origin, endpoint+trunc] and TsdfMap::visit/applyBandUpdate has NO upper gate — front voxels with sdf>trunc are clamped to +trunc and integrated (matching scovoxmap.cpp:298-299 `space_carving && sdf > trunc`). *Suggested fix.* In integrateHitFused, gate the TSDF update the same way applyBandUpdate already does (it only needs `sdf > -trunc`); drop the `sdf <= trunc + h` upper bound, or make it conditional on !space_carving to match the non-fused/legacy band semantics.

### #8 — [`src/scovox_core/include/scovox/binary_serializer_v4.hpp:203`](src/scovox_core/include/scovox/binary_serializer_v4.hpp#L203)  ·  🟠 MEDIUM · CONFIRMED · memory-safety

**reserve() is called with an attacker-controlled u32 count (tsdf_count/beta_count/dir_count) before any payload bytes are validated, allowing a tiny truncated frame to request a multi-gigabyte allocation.**

*Failure scenario.* A 25-byte v4 frame with valid MAGIC/VERSION/header but dir_count field = 0xFFFFFFFF reaches f.dir_deltas.reserve(0xFFFFFFFF). With sizeof(DirDelta)~28B that requests ~120 GB, throwing std::length_error/std::bad_alloc (not the documented 'truncated frame' runtime_error) or thrashing memory. ### #9 — [`src/scovox_core/include/scovox/binary_serializer.hpp:228`](src/scovox_core/include/scovox/binary_serializer.hpp#L228)  ·  🟠 MEDIUM · CONFIRMED · memory-safety

### #10 — [`src/scovox_core/include/scovox/consensus_merge_v4.hpp:55`](src/scovox_core/include/scovox/consensus_merge_v4.hpp#L55)  ·  🟠 MEDIUM · CONFIRMED · numerical

**mergeBeta/mergeDir/mergeSemDir subtract the prior unconditionally with no lower clamp, so a source carrying less-than-prior mass yields negative Beta/Dirichlet alphas, which break p_occ() (outside [0,1]) and downstream entropy/EIG.**

*Failure scenario.* A corrupted, partially-reset, or maliciously crafted v4 wire frame delivers a BetaVoxel with a_occ slightly below occ_prior = C*alpha_0 (e.g. a_occ = 0). isPriorBeta uses a 'v.a_occ <= occ_prior + slop' test, so this voxel is treated as NOT at prior only if a_free exceeds prior; if it is folded as the second source, mergeBeta computes f.a_occ = a.a_occ + 0 - occ_prior < 0. p_occ() = a_occ/(a_occ+a_free) then returns a negative or >1 value, and expectedInformationGain/entropy on that voxel produce NaN/garbage that propagates into ScoreCandidates and GetRegion responses. Same for mergeDir/mergeSemDir when other or a slot cnt is below its prior.

*Suggested fix.* After each subtraction clamp the result to a small positive floor (e.g. f.a_occ = std::max(min_alpha, a.a_occ+b.a_occ-occ_prior)) or validate incoming source voxels are at-or-above prior before folding. Mirror the same guard for f.a_free, f.other, and the slot cnts.

### #11 — [`src/scovox_mapping/include/scovox/node_utils.hpp:133`](src/scovox_mapping/include/scovox/node_utils.hpp#L133)  ·  🟠 MEDIUM · CONFIRMED · numerical

### #12 — [`src/scovox_core/src/uncertainty.cpp:87`](src/scovox_core/src/uncertainty.cpp#L87)  ·  🟠 MEDIUM · CONFIRMED · numerical

**semanticEntropy computes Dirichlet DIFFERENTIAL entropy (same unbounded-below trap as entropy()), and additionally consumes the inflated effectiveResidual as an alpha, producing meaningless values on concentrated/weakly-observed voxels.**

*Failure scenario.* A voxel with one heavily-observed class, e.g. sem_cnt={1000,0}, a_unk=0: alphas={1001, effectiveResidual+1}. The Dirichlet differential entropy lmvb + (a0-K)*digamma(a0) - sum_psi goes strongly negative for such a concentrated Dirichlet (differential, not Shannon, entropy). *Suggested fix.* Replace with the discrete categorical (Shannon) entropy over the K observed slots + residual using mean probabilities p_i = alpha_i / a0, which is bounded in [0, ln(K+1)]. Also bound effectiveResidual as in the first finding.

*Failure scenario.* A .topk whose payload is shorter than total bytes: f.read sets both failbit and eofbit, so f.good()==false and f.eof()==true → condition is (true && false)==false. ### #14 — [`src/scovox_mapping/src/scovox_node.cpp:859`](src/scovox_mapping/src/scovox_node.cpp#L859)  ·  🟠 MEDIUM · CONFIRMED · concurrency

**scheduleMemUsage launches a detached thread capturing `this` that can outlive the node (use-after-free on shutdown) and can run concurrently with itself, corrupting the tsdf_dump file.**

*Failure scenario.* With log_mem_usage:=true, every ~10 frames a detached std::thread takes a shared_lock and walks the whole grid. ### #20 — [`src/scovox_mapping/test/test_consensus.cpp:36`](src/scovox_mapping/test/test_consensus.cpp#L36)  ·  🟠 MEDIUM · CONFIRMED · test-gap

**The consensus 'reset-to-prior + refold-all-sources' double-count safeguard (refoldCell in dscovox_node.cpp, and the v4 split refold using isPriorBeta/isPriorDir) is never tested: no test re-sends a snapshot and asserts the fused result is unchanged. mergeFramesV4 is tested but the receiver-side refold that actually prevents double-counting is not.**

*Failure scenario.* A source re-publishes the same v4 snapshot twice (the exact scenario the architecture says 'never double-count'). The receiver must reset fused[c] to prior and re-fold the source's current value, so the fused state is identical after the second receipt.
